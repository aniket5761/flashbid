FlashBid Project Report
=======================

Project Name
------------
FlashBid

Project Type
------------
Full-stack real-time auction platform with a Spring Boot backend, React frontend, PostgreSQL persistence, Redis-backed live state/event delivery, and JWT-based authentication.


1. Executive Summary
--------------------

FlashBid is a web application built to support time-bound product auctions where users can browse items, place bids, monitor auction progress, and receive live auction updates as bidding activity happens. The platform is designed to handle both standard CRUD-style application behavior and event-driven real-time behavior. This is important because auction systems are different from ordinary product catalog systems: data changes are frequent, timing is critical, bid validation must be strict, and the user interface must react quickly when the current highest bid changes.

The project is split into two major parts:

1. A Spring Boot backend that owns business logic, authentication, persistence, bidding rules, auction status transitions, and websocket broadcasting.
2. A React frontend that renders the user experience, consumes REST APIs, connects to live auction topics, and updates the UI when new bids or auction status changes happen.

The system also uses:

1. PostgreSQL for relational storage of users, products, bids, auctions, and auction winners.
2. Redis for caching recent live auction state and distributing live events through pub/sub.
3. STOMP over SockJS for browser-friendly live websocket communication.

This architecture allows the platform to support both strong business validation and responsive user experience. REST handles secure, traceable operations such as placing bids and managing users. Websockets handle live updates so the UI does not need to constantly poll the server.


2. Product Goals
----------------

The main goals of FlashBid are:

1. Let users discover active and scheduled auctions.
2. Allow authenticated users to place bids safely.
3. Prevent invalid bids and unauthorized bidding.
4. Show current bid state in real time.
5. Automatically open and close auctions based on time.
6. Determine auction winners correctly after closing.
7. Support role-based access for regular users, sellers, and admins.
8. Provide a modern frontend that works well on desktop and mobile.
9. Make deployment practical using common cloud hosting patterns.
10. Preserve data consistency even when multiple bidders act at nearly the same time.

4. Backend Architecture
-----------------------

The backend is built with Spring Boot. It is organized into domain modules and shared infrastructure.

Important backend modules:

1. `auth`
   Handles registration, login, JWT creation, and user identity lookup.

2. `user`
   Handles user data, roles, moderation, and profile-related operations.

3. `product`
   Handles product CRUD behavior, product DTO mapping, and listing/filtering.

4. `bid`
   Handles bid submission and bid history retrieval.

5. `auction`
   Handles auction status, time-based open/close behavior, and winner determination.

6. `common`
   Holds security config, websocket config, Redis config, exception handling, and shared utility logic.

Backend request flow:

1. The controller receives the HTTP request.
2. Spring Security authenticates the request if needed.
3. The service layer applies business rules.
4. Data is stored or retrieved through repositories.
5. Live update services refresh cached auction state and publish events.
6. Websocket subscribers receive updated auction state through STOMP topics.

This layered approach keeps responsibilities separate and maintainable.


5. Frontend Architecture
------------------------

The frontend is a React application built with Vite.

Important frontend areas:

1. `pages`
   Route-level screens such as Home, Products, ProductDetail, Profile, SellerStudio, and AdminDashboard.

2. `component`
   Shared UI pieces like cards, fields, pagination, headers, and badges.

3. `api`
   Axios client setup, format utilities, and websocket client code.

4. `state`
   Authentication state management via context.

Frontend flow:

1. React Router matches a route.
2. The page component loads initial data using Axios.
3. If the page is live-enabled, it creates a SockJS/STOMP client.
4. Websocket messages update local component state.
5. UI rerenders with current auction state.

Why React was chosen:

1. Component-based structure suits dashboard and marketplace UIs.
2. State-driven rendering works well for live-updating views.
3. React Router provides clean route management.
4. Vite gives fast developer feedback and simple deployment.
5. React’s ecosystem supports maintainable form handling, API integration, and progressive scaling.


6. Why Spring Boot
------------------

Spring Boot was a strong choice for this project for several reasons.

1. It is excellent for structured backend applications with authentication, validation, data access, and scheduling.
2. The project needs REST endpoints, websocket support, security filters, dependency injection, scheduling, and database integration. Spring Boot supports all of these in a unified ecosystem.
3. Spring Security makes JWT-based route protection practical.
4. Spring Data JPA reduces boilerplate for entity persistence.
5. Spring scheduling is useful for time-based auction state transitions.
6. Spring’s transaction support is very valuable in bidding systems where concurrent writes can happen.
7. WebSocket and STOMP support are mature and integrate well with the rest of the stack.

Why not a lighter backend?

1. Auction systems quickly become business-rule heavy.
2. The project needed more than simple CRUD.
3. Transaction boundaries, security, and integration consistency matter a lot.
4. Spring Boot gives a strong foundation for systems that may grow in complexity.


7. Why PostgreSQL
-----------------

PostgreSQL was chosen because auction platforms have relational data with integrity requirements.

Reasons:

1. Users, products, auctions, bids, and winners have clear relationships.
2. Transactions are critical when recording bids.
3. Strong consistency is important to prevent incorrect auction state.
4. SQL querying is useful for filtering, pagination, sorting, and reporting.
5. PostgreSQL is reliable, widely supported, and works well with Spring Data JPA.

Why PostgreSQL fits better than a document database here:

1. Bid history is relational and ordered.
2. Auction winner computation depends on consistent bid data.
3. Product ownership and user roles fit relational modeling naturally.
4. ACID guarantees matter in competitive bidding flows.


8. Why Redis
------------

Redis is used for two major purposes:

1. Cache support for live auction summaries and recent bids.
2. Pub/sub support for distributed live event broadcasting.

Why Redis helps:

1. It is very fast for frequently changing, lightweight live data.
2. It can store temporary derived auction state without overloading PostgreSQL.
3. Redis pub/sub allows one part of the backend to publish a live event and another to broadcast it to websocket subscribers.
4. It improves responsiveness when many users are viewing the same auction.
5. It supports scaling better than keeping everything only in one process memory.

Without Redis:

1. Every live update would require rebuilding the same state from the database.
2. Multi-instance deployment would be harder for event distribution.
3. UI responsiveness under load would degrade faster.


9. Why WebSockets Instead of Polling
------------------------------------

Auction systems need timely UI updates. Polling every few seconds is inefficient and can still feel stale.

Why websockets are better here:

1. Users receive updates immediately after a bid is placed.
2. The frontend does not need to repeatedly call the same APIs.
3. Server push gives a more auction-like experience.
4. It reduces redundant request load on the backend when many users watch the same product.

Why STOMP over SockJS:

1. STOMP gives a structured topic/subscription model.
2. SockJS improves compatibility and fallback behavior for browsers and proxies.
3. It fits the `/topic/auctions/{productId}` model cleanly.


10. Data Model Overview
-----------------------

Main entities:

1. User
   Stores identity, role, profile data, and moderation-related fields.

2. Product
   Represents the auctioned item and seller ownership.

3. Auction
   Stores auction-specific timing and increment settings.

4. Bid
   Stores each bid amount, bidder identity, timestamp, and product relationship.

5. AuctionWinner
   Stores the winning bidder and winning amount after closure.

Supporting live state models:

1. AuctionSummaryCache
   Stores summary fields such as current bid, next minimum bid, bid count, and top bidder.

2. AuctionLiveEvent
   Carries live update payloads to the frontend.


11. Auction Lifecycle
---------------------

An auction moves through three major states:

1. SCHEDULED
2. OPEN
3. CLOSED

Lifecycle behavior:

1. Before `startTime`, the auction is scheduled.
2. At or after `startTime` and before `endTime`, it is open.
3. At or after `endTime`, it is closed.

The backend scheduler runs periodically and updates auction states. This avoids depending only on user actions to move the auction forward.

When an auction closes:

1. Backend resolves the winner.
2. Product status is updated.
3. A live update event is broadcast.
4. Winner information becomes available through REST and websocket updates.


12. Bid Validation and Race Condition Handling
----------------------------------------------

Race conditions are one of the biggest risks in an auction application. Two users may try to place bids at nearly the same moment. If the backend does not control concurrency carefully, both bids may incorrectly pass validation against the same old highest amount.

How the project addresses this:

1. Bid placement happens in a transactional service method.
2. The auction row is loaded using a locking query (`findByProductIdForUpdate`).
3. Current highest bid is checked inside the same controlled transaction.
4. Minimum bid amount is computed from authoritative backend state.
5. Only then is the new bid saved.

Why this works:

1. The database lock prevents overlapping bid validation from using stale state for the same auction.
2. Validation remains server-side, so the frontend cannot bypass it.
3. The persisted bid becomes the source of truth before the live event is emitted.

This is the correct place to solve race conditions. The frontend can help user experience, but only the backend can guarantee correctness.


13. Security Model
------------------

The application uses JWT-based stateless authentication.

Main security ideas:

1. Public routes are allowed for registration, login, and viewing products.
2. Protected routes require a valid JWT.
3. Role checks control access to admin or seller functions.
4. Unauthorized responses trigger frontend logout behavior.
5. CORS is configured to allow the expected frontend origins.
6. Websocket handshake endpoints are explicitly permitted.

Why JWT is suitable:

1. It works well for stateless APIs.
2. It is simple to use with frontend local storage and request interceptors.
3. It avoids session server overhead in a decoupled frontend/backend architecture.

The project is structured so frontend and backend can be deployed independently.

Typical deployment setup:

2. Backend on Railway.
3. PostgreSQL hosted externally.

This project includes several performance-conscious ideas:

2. Bid history is paginated.
3. Recent live state is cached.

16. At Least 30 Problems That Can Occur and How To Solve Them
-------------------------------------------------------------

Below is a detailed issue list covering design, implementation, deployment, data consistency, frontend behavior, and operations.

Problem 1. Live updates work locally but not in deployment
Solution:
Local profile enabled Redis pub/sub while deployed config defaulted to disabled. Set `APP_REDIS_PUBSUB_ENABLED=true` in deployment and ensure Redis is reachable. Also keep a direct broadcast fallback for single-instance cases if needed.

Problem 2. Websocket connects but UI still does not update
Solution:
Handshake success only proves the socket opened. Verify the backend is actually publishing auction events and that the Redis subscriber forwards them to `/topic/auctions/{productId}`.

Problem 3. CORS blocks requests from the deployed frontend
Solution:
Set `APP_ALLOWED_ORIGINS` to the exact frontend URL, for example `https://flashbid.vercel.app`. Use comma-separated values for multiple origins. Prefer `allowedOriginPatterns` if wildcard-style patterns are needed safely.

Problem 4. REST works but websocket connection fails in deployment
Solution:
Check `VITE_WS_BASE_URL`, reverse proxy behavior, and whether `/ws` is exposed properly. Also ensure the backend allows the websocket origin and that the frontend derives the correct websocket path.

Problem 5. Websocket path breaks when API is behind a path prefix
Solution:
Do not blindly hardcode `/ws` at root if the deployed path structure differs. Derive websocket URL carefully from the configured backend base URL or set `VITE_WS_BASE_URL` explicitly.

Problem 6. Multiple API calls happen when opening product detail
Solution:
Separate product loading from bid loading. Product details should depend on `productId`, while bid history may depend on auth state. Avoid bundling unrelated requests into one effect dependency chain.

Problem 7. UI appears to load twice in development
Solution:
React Strict Mode intentionally double-invokes effects in development. This is normal for dev mode and should not be confused with production behavior. Make effects idempotent.

Problem 8. UI appears to reload twice in production
Solution:
This usually means the page has multiple effects or state changes that retrigger the same request. Inspect dependencies and split fetch responsibilities cleanly.

Problem 9. Self-bidding must be prevented
Solution:
Validate on the backend using authenticated user identity and product ownership. Never rely only on frontend controls.

Problem 10. A bid is accepted even though the auction just ended
Solution:
Check auction time and effective status inside the transaction immediately before saving. Use current backend time as the authority.

Problem 11. Two simultaneous bids both pass validation
Solution:
Use row locking and transactional bid placement. Validate against authoritative current state inside the same transaction.

Problem 12. Bid amount below minimum increment is accepted
Solution:
Compute `minimumAllowed = currentHighest + minimumIncrement` on the backend and reject any lower amount.

Problem 13. Auction status in the database becomes inconsistent
Solution:
Centralize status resolution in a dedicated auction management service. Do not let multiple controllers independently decide state transitions.

Problem 14. Auctions never open automatically
Solution:
Use a scheduler to periodically open scheduled auctions whose start time has passed. Ensure scheduling is enabled and deployment clocks/timezones are correct.

Problem 15. Auctions never close automatically
Solution:
Use a scheduler to close auctions after end time and publish closure updates. Verify scheduler frequency and time comparisons.

Problem 16. Winner is missing after auction closure
Solution:
Create winner determination as part of the closure flow and handle the no-bid case explicitly.

Problem 17. Redis connection exists but subscriber never starts
Solution:
The Redis message listener container is often conditional on configuration. Make sure `APP_REDIS_PUBSUB_ENABLED=true` is set and startup logs confirm listener initialization.

Problem 18. Frontend shows stale current bid after a new bid
Solution:
Update UI state from websocket payloads immediately and ensure the payload includes current bid, next minimum bid, and bid count.

Problem 19. Product list view does not update when an auction changes
Solution:
Subscribe to all visible product topics and patch the relevant product in component state when live events arrive.

Problem 20. Too many database reads for live data
Solution:
Cache summary and recent bids in Redis and reuse them when building live events instead of reconstructing everything from PostgreSQL every time.

Problem 21. Authentication token expires but UI still behaves like user is logged in
Solution:
Check token expiry on startup and during app lifecycle, then clear local state and storage when the token is invalid.

Problem 22. User gets stuck after backend returns 401
Solution:
Use an Axios interceptor to trigger logout behavior globally when unauthorized responses are received.

Problem 23. Frontend works locally but fails after deployment because API URL is wrong
Solution:
Set `VITE_API_BASE_URL` in frontend deployment. Local proxy behavior in Vite does not exist automatically in production.

Problem 24. Backend tests fail because datasource env vars are missing
Solution:
Provide test-specific configuration or a test profile. Do not depend on production cloud datasource env vars for local test execution.

Problem 25. Race condition between scheduler status update and incoming bid
Solution:
Always resolve auction status again inside the bid transaction. Scheduler helps move system state, but bid placement must recheck current validity.

Problem 26. Admin user can accidentally bid
Solution:
Disallow admin bidding in backend rules if required by business policy and hide bid controls in frontend for admin users.

Problem 27. Deleted or banned users cause null or confusing UI labels
Solution:
Normalize display labels, for example showing `Deleted seller` or `Deleted account` instead of assuming full user records are always active.

Problem 28. Bid history leaks to unauthenticated users when that is not desired
Solution:
Protect bid history endpoint access rules and gate frontend display according to authentication state.

Problem 29. Auction detail page blocks too long on initial load
Solution:
Load product details first and bid history separately. This reduces perceived latency and prevents unrelated fetch retries from affecting the main auction display.

Problem 30. Frontend websocket stays connected after route change
Solution:
Deactivate the STOMP client in effect cleanup to prevent duplicate subscriptions and memory leaks.

Problem 31. Duplicate live subscriptions cause repeated UI updates
Solution:
Ensure websocket client setup runs once per product subscription scope and always unsubscribes or deactivates on cleanup.

Problem 32. Server cannot distinguish frontend origins cleanly in deployment
Solution:
Keep frontend and backend base URLs explicit in environment variables and document them clearly. Do not rely on implicit current-origin behavior across separate deployments.

Problem 33. Product details endpoint is refetched unnecessarily after auth changes
Solution:
Make product loading depend only on product id. Only auth-sensitive resources should depend on auth state.

Problem 34. Winner information is not shown on live closure
Solution:
Include winner information in the live event payload after closure and update the frontend winner state from that payload.

Problem 35. Auction state and product state drift apart
Solution:
Update both auction status and product status through one controlled service method so they stay aligned.

Problem 36. Deployment works on one instance but breaks after scaling horizontally
Solution:
Use Redis pub/sub for cross-instance event fanout. In-memory-only broadcasting is not sufficient across multiple instances.

Problem 37. SockJS connection succeeds but subscriptions receive nothing
Solution:
Check that the backend sends to the same topic path the frontend subscribes to, such as `/topic/auctions/{productId}`. Topic mismatch is a common issue.

Problem 38. Localhost origin works but deployed frontend still gets blocked
Solution:
Replace default localhost-only origin config with deployment frontend URL. Defaults are helpful for development, not production.

Problem 39. Auction close time behaves differently across environments
Solution:
Use a consistent application timezone and document it. Also make sure stored timestamps and displayed timestamps are formatted clearly.

Problem 40. UI reports a successful bid but numbers do not change immediately
Solution:
Do not rely only on POST response. Let websocket event update the visible live state, and ensure the live payload is actually emitted after transaction commit.


17. How Major Problems Were Solved in This Project
--------------------------------------------------

Below are some especially important solved areas.

A. Live update failure in deployment
-----------------------------------

Root cause:
The application used Redis pub/sub for event fanout, but deployed config did not enable the Redis subscriber. Local worked because local profile enabled it.

Fix:

1. Set `APP_REDIS_PUBSUB_ENABLED=true` in deployment.
2. Verify `SPRING_DATA_REDIS_URL`.
3. Keep backend broadcasting logic aligned with actual deployment topology.

B. Websocket URL mismatch
-------------------------

Root cause:
Frontend websocket path derivation can break when deployment uses a different base URL or path layout.

Fix:

1. Derive websocket URL from configured env values more carefully.
2. Support `VITE_WS_BASE_URL`.
3. Avoid assuming the backend is always on the same origin and root path.

C. Duplicate product detail fetches
-----------------------------------

Root cause:
Frontend page startup mixed product loading and auth-dependent bid loading together.

Fix:

1. Load product by `productId`.
2. Load bids by `productId + auth state`.
3. Prevent repeated identical request sequences.

D. Bid race condition risk
--------------------------

Root cause:
Auction systems can receive overlapping bid requests.

Fix:

1. Transactional bid placement.
2. Locked auction row retrieval.
3. Server-side validation of current highest bid and minimum increment.

19. Lessons Learned
-------------------

This project demonstrates several important engineering lessons.

1. Real-time systems usually fail at boundaries, not in the happy path.
   The main issues were not “can we open a websocket” but “does deployment configuration allow events to travel end-to-end.”

2. Local success can hide production-specific bugs.
   Profile-specific config differences caused the live update issue.

3. Business correctness belongs in the backend.
   Race condition prevention, authorization, and minimum bid validation must not depend on the frontend.

4. Frontend side effects need careful dependency design.
   Duplicate loads often come from state/effect structure rather than network or backend issues.

5. Deployment documentation is part of the system.
   Missing one env variable can make a large feature appear broken.

21. Conclusion
--------------

FlashBid is more than a basic marketplace project. It combines role-based access, transactional bidding, real-time event delivery, auction lifecycle scheduling, and deployment-aware configuration management. That makes it a meaningful example of a full-stack application where correctness, responsiveness, and infrastructure concerns all matter.

The project shows why:

1. Spring Boot is a strong fit for rule-heavy transactional systems.
2. PostgreSQL is a strong fit for relational and consistent auction data.
3. Redis is valuable for live state and event distribution.
4. React is a strong fit for dynamic, real-time, role-aware user interfaces.

It also shows that many of the hardest problems in real systems are not only in writing features, but in making those features correct under concurrency, consistent across environments, and smooth in the user interface.

This combination of architecture, business logic, live delivery, and troubleshooting makes FlashBid a strong practical full-stack project.
